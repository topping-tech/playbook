گزارش اجرایی ارتقای زیرساخت سرور Barzin Holding

خلاصه کامل کارهایی که انجام شد، ارتقاهای ایجادشده، فایل‌ها، کدها و لینک‌های ساخته‌شده

تاریخ: 2026-06-06

خلاصه مدیریتی

در این عملیات، سرور از حالت پراکنده و وابسته به حافظه فردی، به یک زیرساخت منظم، مستند، قابل بکاپ‌گیری، قابل بازیابی و قابل مانیتورینگ تبدیل شد. تمرکز کار روی پروژه‌های CRM، Veloxa، Gozaresh، NFC و AI API بود.

نتیجه اصلی: اکنون برای Barzin Holding یک مرکز کنترل زیرساخت، سیستم بکاپ خودکار، مستندات بازیابی، امنیت پایه و داشبورد مانیتورینگ Live ایجاد شده است.

1. ساختار مرکزی ایجادشده

یک مسیر مرکزی برای نگهداری مستندات، بکاپ‌ها، اسکریپت‌ها، لاگ‌ها و اطلاعات عملیاتی ساخته شد:

/www/holding-barzin
بخشکاربرد

databasesآرشیو بکاپ دیتابیس‌ها و schemaها
docsمستندات، نقشه سرور، Recovery Plan و گزارش‌ها
scriptsاسکریپت‌های بکاپ، Cleanup و تولید وضعیت Live
logsلاگ‌های بکاپ، Cleanup و وضعیت‌ها

2. دیتابیس‌ها و بکاپ‌ها

دیتابیس‌های اصلی شناسایی، مرتب و برای هرکدام پوشه اختصاصی ایجاد شد.

3. مشکل فنی بکاپ و راه‌حل

مشکل اصلی بکاپ این بود که PostgreSQL سرور نسخه 18 بود ولی ابزار پیش‌فرض pg_dump نسخه 14 بود. نتیجه این اختلاف نسخه، ساخت فایل‌های بکاپ صفر بایت بود.

مسیر صحیح پیدا شد و برای بکاپ‌گیری استفاده شد:

/www/server/pgsql/bin/pg_dump

نتیجه: بکاپ‌های واقعی و قابل استفاده تولید شدند.

4. بکاپ خودکار و Cleanup

برای جلوگیری از وابستگی به بکاپ دستی، اسکریپت بکاپ خودکار ساخته و به Cron وصل شد.

عملیاتزمانتوضیح
Backup03:00 AMبکاپ روزانه دیتابیس‌ها
Cleanup04:00 AMحذف بکاپ‌های قدیمی‌تر از 30 روز

فایل‌های اصلی:

/www/holding-barzin/scripts/backup-all-databases.sh
/www/holding-barzin/scripts/cleanup-old-backups.sh

5. امنیت سرور

  • انجام شد PostgreSQL فقط روی 127.0.0.1 فعال شد.
  • انجام شد پورت عمومی 5432 از UFW حذف شد.
  • انجام شد Fail2Ban بررسی شد و فعال بود.
  • انجام شد فایل‌های حساس مثل .env برای CRM، Veloxa و Gozaresh مسدود شدند.
  • تست شد درخواست به /.env خروجی 403 Forbidden داد.

6. PM2 و بازیابی بعد از ریبوت

سرویس‌های Node.js با PM2 ذخیره شدند و Startup فعال شد تا بعد از ریبوت سرور، سرویس‌ها خودکار بالا بیایند.

10. نکته امنیتی مهم درباره لینک آنلاین

داشبورد فعلاً روی لینک CRM قابل مشاهده است و اطلاعات کلی زیرساخت را نشان می‌دهد. این داشبورد نباید شامل پسورد، API Key، Connection String یا اطلاعات محرمانه باشد.

پیشنهاد امنیتی برای مرحله بعد: اضافه کردن Basic Auth یا Cloudflare Access برای محدود کردن دسترسی.

11. خروجی نهایی عملیات

در پایان این کار، زیرساخت Barzin Holding به وضعیت زیر رسید:

  • مرتب‌سازی مرکزی همه چیز در مسیر /www/holding-barzin متمرکز شد.
  • بکاپ خودکار دیتابیس‌ها هر روز ذخیره می‌شوند.
  • امنیت پایه PostgreSQL، UFW، Fail2Ban و .env Protection بررسی و تنظیم شدند.
  • بازیابی Disaster Recovery Plan و PM2 Recovery ساخته شدند.
  • مانیتورینگ داشبورد Live برای بررسی وضعیت سرویس‌ها ساخته شد.
  • مستندسازی نقشه سرور، مسیرها، سرویس‌ها و فایل‌ها ثبت شدند.

جمع‌بندی رسمی

این عملیات، پایه یک Infrastructure Control System برای Barzin Holding را ایجاد کرد. اکنون سرور قابل بررسی، قابل بازیابی، دارای بکاپ خودکار، دارای مستندات و دارای داشبورد مانیتورینگ Live است.

این سند برای نگهداری داخلی، تحویل به مدیر IT، بررسی مدیریتی و ارزیابی وضعیت فنی سرور قابل استفاده است.
